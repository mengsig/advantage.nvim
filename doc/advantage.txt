1. INTRODUCTION                                       *advantage-introduction*

advantage.nvim is a coding-agent *harness* that runs its own agent loop inside
Neovim: it streams from the model, executes tools in your editor, feeds the
results back, and repeats until the task is done. It is not a wrapper around a
CLI.

- Runs on your subscription: Claude Code login (Pro/Max) or Codex / ChatGPT
  login. Environment API keys work as a fallback.
- Model-agnostic: Anthropic and OpenAI/Codex families out of the box.
- Editor-native tools executed inside Neovim, all edits reviewable.

==============================================================================
2. REQUIREMENTS                                       *advantage-requirements*

- Neovim >= 0.10
- `curl` on your PATH
- One of: a `claude` CLI login, a `codex` CLI login, or `$ANTHROPIC_API_KEY` /
  `$OPENAI_API_KEY`.
- Optional: `rg` (ripgrep) for fast grep/file listing; `wl-paste`/`xclip`/
  `pngpaste` for clipboard image paste.

4. COMMANDS                                               *advantage-commands*

                                                                   *:Advantage*
:Advantage              Toggle the chat panel.
:Advantage new          Start a new session.
:Advantage model        Pick the model.
:Advantage resume       Resume a saved session for this project.
:Advantage stop         Cancel the in-flight turn.
:Advantage usage        Token usage dashboard.
:Advantage compact [llm|heuristic]
                        Force conversation compaction. Defaults to
                        `context.compact_mode`; "llm" spends one call on
                        `context.summarizer_model` for a real summary,
                        "heuristic" stays free/offline. Messages submitted
                        meanwhile queue and send after compaction completes.
:Advantage context      Show/manage per-repo memory (see |advantage-memory|).
:Advantage review       Review the agent's file changes (diff).
:Advantage yolo [on|off]
                        Toggle skip-all-permissions mode.
:Advantage effort [mode]
                        Tune reasoning effort / thinking.
:Advantage harness [mode]
                        Tune harness orchestration mode.
:Advantage help         Keybinding & command cheatsheet.
:Advantage add          Send the current file / selection to the prompt
                        (with a range, e.g. `:'<,'>Advantage add`, sends
                        `@file:L10-20`).
:Advantage files        Pick a project file to add to the prompt.
:Advantage attach {path}
                        Attach a file (images become attachments, other files
                        `@mentions`). No path opens the file picker.
:Advantage ask {prompt} Send a prompt directly. With a visual range
                        (`:'<,'>Advantage ask why is this slow?`) the selected
                        lines are appended as a fenced block.

BUILT-IN ENGINEERING POLICY                         *advantage-system-prompt*

The default system prompt is language-agnostic and carries ten ordered
commandments: write for humans, keep units cohesive, enforce invariants, expose
failures, make data flow and ownership explicit, fit the design to the problem,
verify behavior, preserve clean boundaries, model state soundly, and keep changes
narrow and compatible. Tool-, memory-, verification-, and harness-specific
instructions are separate conditional blocks. Configure `system_prompt` with a
string to replace the default or `function(default)` to extend it. Use
`context preview` to inspect the exact composed prompt and per-part token cost.

THINKING AND REASONING                              *advantage-effort-levels*

`:Advantage effort [mode]` and `/effort [mode]` only offer values supported by
the active model and OpenAI transport:

- Modern Claude models use adaptive thinking plus `output_config.effort`
  (`low` through `max`). Opus 4.8 is explicitly adaptive; Sonnet 5 is default-on
  and uses an explicit disabled state for `off`; Fable 5 cannot be disabled.
- Haiku 4.5 and older manual-thinking models use fixed budgets from 1k through
  32k. Thinking and visible output share `providers.anthropic.max_tokens`.
- OpenAI uses literal `reasoning.effort` values through `max`. Ultra is a
  separate harness mode, not a wire-level effort. Raw API profiles use their separate
  `api_reasoning_efforts`. `off` means explicit `none` and is only available
  where supported. Inherited values clamp to the deepest supported wire level;
  explicit unsupported model overrides remain errors.

HARNESS MODES                                         *advantage-harness-modes*

`:Advantage harness [mode]`, `/harness [mode]`, and `<leader>ch` control how
the parent agent orchestrates work. Selecting a preset initializes matching
model effort; `/effort` may then override reasoning independently. `auto`
derives the policy from the current effort.

  low      direct parent work; sequential scouts
  medium   selective delegation with modest concurrency
  high     proactive splitting of clearly independent investigations
  xhigh    deep parent reasoning with selective, task-sized fan-out
  max      maximum parent depth; deliberate delegation
  ultra    Max reasoning; proactive parallelism when work divides cleanly

`subagents.parallel = false` is a hard global override and keeps every mode
sequential. `max_parallel` controls scheduling width, never how many scouts may
run. XHigh/Max/Ultra permit an explicitly configured width up to eight while the
safe default remains four.

OPENAI TRANSPORT                                    *advantage-openai-transport*

`providers.openai.auth_mode = "auto"` prefers a usable Codex/ChatGPT login and
falls back to an API key. Use `"chatgpt"` or `"api_key"` to require one path.
The paths can expose different model catalogues and limits; built-in GPT-5.6
models budget 372k context tokens for login and 1.05M for the raw API (GPT-5.5:
272k login / 1.05M API). GPT-5.6/5.5 login requests reserve their native 128k
output maximum for context safety; raw API requests send and reserve the
configured 64k cap. OpenAI requests
route identical system/tool prefixes through a stable content-derived
prompt-cache key across chats. On ChatGPT login, each chat separately keeps a
unique stable session id across tool-loop turns. Scout prefixes use four
round-robin cache buckets so concurrent fan-out retains reuse without one hot
route.

`providers.openai.reasoning_summary = "auto"` streams a provider-selected
reasoning summary into the thinking UI. `"concise"` and `"detailed"` request a
specific depth. Set it to false to omit summaries; scouts and compaction
summarizers disable them automatically.

Pre-payload OpenAI overload and truncated-stream events retry the exact request
with bounded backoff and a visible notice. Once text, thinking, or a tool call
has streamed, no retry is allowed, preventing duplicated work. Authentication,
model, and deterministic request errors remain non-retryable.

SUB-AGENT EXECUTION                                  *advantage-subagents*

Sub-agents are read-only scouts. The model may delegate one task and wait, or
emit a same-turn batch. With `subagents.parallel = true`, batch members run
concurrently; this advertises parallel support but never forces the model to use
it. Scouts cannot run shell commands, create fixtures, or execute a CLI; runtime
testing remains parent work. Set it false for sequential execution. There is no per-batch or cumulative
spawn quota: every valid scout the parent requests runs. `max_parallel` is only
the concurrency width; excess scouts wait in a local queue. Independent scouts
within a mixed tool response run as one concurrent contiguous group while result
ordering remains unchanged. The default policy uses one task-sized investigation
wave, then returns to parent synthesis and implementation. Later waves remain
available for a concrete new blocker; generic surveys and post-implementation
review waves are discouraged, not rejected. Sequential batches only serialize
self-contained prompts; data-dependent scouts belong in separate parent turns.
Results share the
`max_result_bytes` context budget. `max_output_tokens` caps scout requests on
Anthropic/raw-API transports. ChatGPT-login scouts retain the model's native
output reserve and are bounded by the individual turn, result, and report caps.
Scouts are steered under roughly 900 words and each returned report is hard-capped
at 6,000 bytes. The ordinary budget is six provider requests, including the
final report-only request, with a configurable hard ceiling of twelve for one
deep blocker. A request-four sufficiency checkpoint asks a scout to report early
when its evidence is already decisive. Child rows show the provider, route,
effort, and live request n/N progress. Reports separate root cause/evidence, a minimal compatible touch set,
contracts to preserve, focused regression cases, and optional hardening. Parent
phase guidance treats reports as evidence, protects previously passing test
contracts, requires hermetic new tests, and allows one batched source-confirmation
pass before explicitly steering the parent to implement. It tracks successful verification per
edit generation. A later edit re-arms one bounded final diff audit; repeating a
test without edits does not repeat the reminder.
Every call explicitly names a stable short alias and effort level. By default,
Codex/OpenAI parents expose only sol/terra/luna and Claude/Anthropic parents
expose only opus/sonnet/haiku. Set `subagents.allow_cross_provider = true` for a
deliberate mixed-provider wave. `subagents.model_aliases` maps
those choices to current configured provider IDs, keeping fragile version
strings out of model output. Blank/raw first-party IDs are rejected locally.
Provider-auth and unsupported-model failures temporarily remove affected aliases
from later schemas so the parent cannot retry sibling IDs blindly.
`subagents.model` and `subagents.effort` are preferences shown to the parent, not runtime defaults.
The parent may explicitly choose `"inherit"` when intentional.

7. TOOLS & SECURITY                                          *advantage-tools*

Tools: `read_file`, `edit_file`, `write_file`, `multi_edit`, `bash`, `grep`,
`find_files`, `list_dir`, `diagnostics`, `sub_agent`, `sub_agent_batch`,
`web_search`, `web_fetch`, optional `navgraph`, plus memory tools.

File tools are confined to the project root: absolute paths, `..` traversal,
and symlinks that resolve outside the repo are rejected (set
`tools.allow_outside_root = true` to opt out). Mutating tools (`edit_file`,
`write_file`, `multi_edit`, `bash`) require an approval card unless auto-approved
or `yolo` is on. Sub-agents are read-only.

After a mutating edit the newly-introduced LSP/linter diagnostics are appended to
that tool's result (errors only by default, capped, diffed against the pre-edit
state, silent on a clean edit) so the model self-corrects. A no-LSP repo pays no
per-edit cost and instead gets a once-per-filetype "install a language server"
notice â€” persisted as a line in the chat transcript (plus a WARN toast) so you
won't miss it if you stepped away. Configure via `tools.diagnostics` (`enabled`, `auto`, `severity`, `max`,
`wait_ms`, `attach_grace_ms`, `notify_missing`); the model also has an explicit
`diagnostics` tool.

`web_search` uses the Brave Search API when keyed and a best-effort public
Brave/Bing HTML fallback otherwise. `web_fetch` independently retrieves a known
public HTTP(S) page for verification. Fetch is GET-only and DNS-pinned, checks
every redirect, rejects private/metadata hosts, credentials, non-default ports,
binary/oversized responses and HTTPS downgrades, and caps time and returned
lines. Search snippets and fetched pages are wrapped as untrusted web data;
instructions inside a page never become agent instructions. Both tools are safe
(no approval prompt) and available to read-only sub-agents; only stable Brave
API search requires `$BRAVE_API_KEY` or `tools.web_search.api_key`.

NavGraph integration is deliberately opt-in. Set
`tools.navgraph.enabled = true`; the tool appears only after its configured
`executable` passes the `navgraph.capabilities.v1` / agent-protocol handshake.
Advantage negotiates once per stable executable identity, freezes a deterministic
read-only intersection, and fails closed with one diagnostic when the contract
is malformed or incompatible. The pre-contract `84986b8` benchmark binary is
accepted only by its exact SHA-256 and can be disabled with
`allow_legacy_benchmark = false`; arbitrary old binaries are never guessed.
This safe tool is auto-approved, so enabling it
trusts that external executable; prefer a pinned absolute path for reproducible
environments and to prevent PATH shadowing. The parent and read-only scouts
receive the same conservative query-only command set. Each invocation is a
direct argv call (never a shell),
uses the active project as its fixed `-C` root, forces `--no-cache` so a query
cannot write `.navgraph/cache`, rejects `rename`, `serve`/MCP, caller-supplied
root flags, and external/symlink-escaping `read` paths, and stops at the configured
time/output bounds. Git-backed history/diff commands are excluded because local
Git configuration can invoke repository programs. NavGraph's grep-style exit
code 1 is a successful no-match; parse-health warnings remain visible, while
other exit-1 diagnostics are reported as operational errors. Forcing
`--no-cache` means every call reindexes; this deliberately trades latency on
very large repositories for no workspace writes, so avoid redundant parallel
queries.

Current NavGraph advertises a compact typed MCP `navgraph.query` surface. This
release records that capability but still executes the conservative one-shot CLI
adapter described here; typed MCP session/reload/pagination adoption remains a
separate integration step.

Leading-hyphen positional data is exposed only when the executable passes a
`--` terminator probe; for example, `strings` may then search for literal
`--no-tests` without interpreting it as an option. Oversized valid `read`
selections return a deterministic ascending prefix with explicit partial-result
metadata. Indexed commands require and receive `--no-cache`; a command whose
manifest declares both `requiresIndex = false` and `cacheEffect = "none"` may
omit that unsupported flag while remaining inside the read-only intersection.

The tool interface is typed: `target` is one symbol/name pattern/path/filter,
`destination` is the second symbol for `path`, and `limit`, `depth`, `kind`,
`refs`, `strict`, `sort`, `exclude_public`, and `follow_imports` are validated
command options. Raw argv is not accepted. Search is lexical symbol-name search,
not a prose/semantic question. Broad outline/import/route/event requests require
a target/filter, and `read` requires closed `file:A-B[,C-D]` ranges. Overlapping
ranges are merged and their unique source-line total must fit the explicit/live
result limit; open-ended and oversized reads are rejected before spawn.
Discovery returns compact names; `def` automatically returns the indexed symbol
span and full source. The live `max_results` cap is advertised in the schema and
excess requests are rejected instead of silently clamped. An unambiguous, untruncated
definition without a parse-health warning replaces a broad `read_file`; use one
narrow file read only when it is ambiguous, unhealthy, truncated, or needs
surrounding lines. Do not map the same area concurrently in the parent and
scouts, and treat an impact list as evidence rather than an edit list.
Successful results of at least 512 bytes are context-aged: discovery remains
full for one provider request and exact `def`/`read` source for two, after which
the paired tool result becomes a short receipt and provider-private replay state
is detached. Smaller results are already cheap and remain verbatim. Retention
metadata follows copied compacted/session messages, so an unconsumed result is
never hidden before its first reasoning request. Claude extended-thinking tool
continuations are a protocol boundary: their latest signed thinking blocks and
preceding context remain byte-identical until the continuous tool turn closes;
receipt aging and compaction resume afterward.

8. MEMORY & SKILLS                                          *advantage-memory*

advantage keeps per-repo memory under `<root>/.advantage/` in two tiers by cost:

  context.md            ALWAYS-LOADED crisp facts, one bullet each â€” the recurring
                        per-turn tax (rides the cached prefix), held to
                        `memory.budget_tokens` (default 2000). Keep it lean.
  skills/<name>/SKILL.md  DEPTH: a named procedure / deep-dive, index-only in
                        context until loaded on demand via use_skill. The index is
                        itself capped (`memory.skills_index_budget_tokens`) so a
                        large library never re-bloats the prefix.

Because the always-loaded tier is billed every turn, DEPTH belongs in skills, not
in facts: when a fact grows too verbose the `remember` tool steers the agent to
extract it into a skill (with a retrieval-rich description) and leave a crisp
pointer, and persistently nudges you to `/context curate`.

`:Advantage context` shows the memory; `context init` teaches the repo in one
pass; `context curate` compresses it (tighten facts, push depth into skills); `context verify` flags facts whose files
have moved/vanished; `context preview` (also `<leader>cP`) shows the exact
context packet that will be sent â€” system prompt, tool schemas and transcript â€”
with the cache boundary drawn and a per-section token breakdown, so you can see
what each part costs and confirm the memory block is frozen for cache reuse;
`context forget <text>` drops matching facts. Disable with
`memory = { enabled = false }`. Instructions layer from the Git root down to
the launch directory (`AGENTS.override.md`, `AGENTS.md`, `CLAUDE.local.md`,
then `CLAUDE.md` per directory). Skills are discovered from
`.advantage/skills`, `.agents/skills` and `.claude/skills`; use_skill reports
the skill directory so bundled relative references and scripts remain usable.

CUSTOM CONFIG DOCS                                    *advantage-config-docs*

Drop any Markdown file into `<root>/.advantage/` to add your own standing
instructions: every `.advantage/<name>.md` is injected verbatim into the system
prompt each turn (name-sorted for a cache-stable prefix, each capped by
`memory.config_budget_tokens`). No code change and no tool call is needed â€” e.g.
`.advantage/style.md` or `.advantage/review-rules.md`.

  NOTE: `context.md` CANNOT be used this way. It is the managed memory file
  (owned by the remember/curate machinery) and is excluded from config-doc
  ingestion. Put hand-authored instructions in any OTHER `.md` filename.

9. EXTENSIONS                                             *advantage-extensions*

`extensions = { "my_module" }` loads each Lua module once during setup. A module
returns `function(api)` or `{ setup = function(api) ... end }`. The API exposes:

  api.register_tool(def)              add a provider-visible tool
  api.register_provider(name, mod)    add/replace a provider implementation
  api.register_harness(name, policy)  add an orchestration preset
  api.register_prompt_part(name, fn)  add stable session-frozen instructions

All registrations return disposer functions for clean
development reloads. With the default empty list, no extension code runs and
the baseline prompt and schema are unchanged.

Skills may set `advantage-harness: <registered-mode>` in YAML frontmatter.
Activation requires `memory.allow_skill_harness = true`; the default is false.
